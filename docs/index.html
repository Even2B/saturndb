SaturnDB Documentation

A realtime event-sourced database with sub-millisecond push to all subscribers on every write. Append-only event log with Redis-inspired protocol, history replay, and stream-level access control.

⚡ <1ms pub/sub ∞ Append-only log 🔐 Token-based auth 🦀 Built in Rust
⚡

Sub-millisecond Push

Every write fans out to all pattern subscribers using tokio broadcast channels.

📜

Append-only Log

Events are never deleted — history is preserved forever and replayable from any timestamp.

🔍

Pattern Subscriptions

Watch entire namespaces with glob patterns like users:* or orders:*.

🔐

Token-based Auth

Admin, Writer, and Reader roles with per-namespace access control and token expiry.

⚙️

ACID Transactions

Atomic multi-stream writes with BEGIN / COMMIT / ROLLBACK.

🌐

WebSocket Proxy

Browser clients connect on port 7380 — no separate process, embedded in the server.

🔒

Optional TLS

Self-signed cert auto-generated. Toggle tls = true in config for production.

🚀

Fast Startup

Snapshot + WAL replay — restores full state instantly without replaying the entire log.

Getting Started

Get SaturnDB running locally in under two minutes. You'll need the Rust toolchain installed.

1

Build the server

Clone the repository and build in release mode for best performance.

bash
git clone https://github.com/your-org/saturn.git
cd saturn
cargo build --release
2

Start the server

The server reads saturn.toml from the current directory and listens on port 7379 (TCP) and 7380 (WebSocket).

bash
./target/release/saturn-server
3

Use the CLI

The saturn CLI reads credentials from environment variables. Set SATURN_TOKEN and start sending commands.

bash
export SATURN_TOKEN=saturn-admin-secret
export SATURN_HOST=127.0.0.1
export SATURN_PORT=7379

./target/release/saturn ping
./target/release/saturn emit users:1 '{"name":"alice","online":true}'
./target/release/saturn get users:1
./target/release/saturn keys "*"
./target/release/saturn watch "users:*"
4

Connect with an SDK

All SDKs follow the same SaturnDBClient(host, port, token) interface with automatic auth and reconnection.

javascript
const { SaturnDBClient } = require('./sdk/js/saturn.js');

const client = new SaturnDBClient('127.0.0.1', 7379, 'my-token');
await client.connect();

// Write an event
await client.emit('users:1', { name: 'alice', online: true });

// Read latest state
const user = await client.get('users:1');

// Subscribe to a pattern
await client.watch('users:*', (stream, data) => {
  console.log(`[${stream}]`, data);
});

// Replay full history
const history = await client.since('users:1', 0);

Security

JWT Authentication

Saturn can verify Supabase (or any HS256) JWTs directly. Once configured, clients pass their JWT as the AUTH token — no separate token management needed.

toml
[auth.jwt]
secret     = "your-supabase-jwt-secret"   # Settings → API → JWT Secret
role       = "writer"
namespaces = ["user:{sub}", "ride:{sub}"]  # {sub} replaced with user_id

The client authenticates with their Supabase JWT exactly like any other token:

text
AUTH eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
OK

Saturn verifies the signature, extracts the sub claim (user UUID), and creates an ephemeral session scoped to the configured namespaces. The session inherits the JWT's exp — it auto-expires when the JWT expires.

Use {sub} in namespace templates to automatically scope each user to their own data: namespaces = ["user:{sub}"] means user abc-123 can only access streams starting with user:abc-123.

Stream Ownership

CLAIM locks a stream to the authenticated user. After claiming, only the owner (or an admin) can EMIT, PATCH, or DEL to that stream. Reads are still governed by namespace access.

text
# Authenticated as JWT user (sub = "abc-123")
CLAIM driver:abc-123
OK claimed driver:abc-123 for abc-123

# Another user trying to write → rejected
EMIT driver:abc-123 {"lat":1.23,"lng":4.56}
ERR forbidden: stream is owned by another user

Ownership is persisted in the snapshot and survives server restarts. Requires a JWT session — static tokens without a user_id cannot claim streams.

Ride-hailing pattern: when a driver connects, have them CLAIM driver:{their_id}. Now only that driver can update their own location stream — no other client can spoof their position.

Token Templates

Token templates auto-create access tokens whenever a matching stream is written. This is the equivalent of row-level security for programmatic token issuance — no edge function needed.

text
# When any user:* stream is written, create a scoped token for that user
WHEN TOKEN user:* rider:$stream_id writer ride:$stream_id,driver:$stream_id
OK token rule added: user:* → rider:$stream_id

# Now emit a user stream
EMIT user:123 {"name":"evan"}
OK

# Token rider:123 is now automatically available with access to ride:123 and driver:123
VariableValue
$streamFull stream name, e.g. user:123
$stream_idPart after the last :, e.g. 123
$tsEvent timestamp (Unix microseconds)
$<field>Any top-level JSON field from the event payload

Manage token template rules with:

text
WHEN TOKEN LIST
TOKEN_RULE user:* rider:$stream_id writer ride:$stream_id,driver:$stream_id

WHEN TOKEN DEL user:* rider:$stream_id
OK token rule removed: user:* → rider:$stream_id

Rules are persisted to saturn-rules.json alongside WHEN EMIT rules and survive restarts.

Security Hardening

Failed Auth Limit

Saturn drops a connection after max_auth_failures consecutive bad AUTH attempts. The counter resets on a successful auth. Default: 5.

toml
[server]
max_auth_failures = 5   # 0 = disabled

Per-IP Connection Rate Limit

Limits new TCP connections per source IP using a token bucket. Protects against connection-flood attacks. Default: 10/sec, burst 20.

toml
[server]
max_conn_per_ip_per_sec = 10.0  # 0 = disabled
max_conn_burst_per_ip   = 20.0

Audit Log

An append-only JSONL file that records every write operation. Fire-and-forget — never blocks request handling.

toml
[storage]
audit_log = "saturn-audit.log"

Each line is a JSON object:

json
{"ts":1712345678,"op":"EMIT","user":"abc-123","stream":"driver:abc-123","result":"OK"}
{"ts":1712345679,"op":"AUTH","user":"-","stream":"-","result":"ERR"}
{"ts":1712345680,"op":"CLAIM","user":"abc-123","stream":"driver:abc-123","result":"OK"}
Operation loggedDescription
AUTHAll auth attempts (success and failure)
EMIT, PATCH, DELAll write operations
CLAIMStream ownership changes
TOKEN_CREATE, TOKEN_REVOKEToken management

Durable Subscriptions

WATCH drops events if you disconnect. SUBSCRIBE is the reliable alternative — it replays everything you missed then streams live events, so you never lose a message across reconnects.

How it works

  1. Client sends SUBSCRIBE <pattern> FROM <last_event_id>
  2. Server replies SUBSCRIBED <pattern> <count>
  3. Server streams SEVENT <stream> <id> <payload> for each historical event (id > last_event_id), sorted by id
  4. Server sends LIVE — end of replay, now streaming in real-time
  5. All subsequent events arrive as SEVENT (same format, includes id)

The client persists the last id it received. On reconnect it sends FROM <that_id> and picks up exactly where it left off.

Example

text
# First session — subscribe from beginning
SUBSCRIBE orders:* FROM 0
SUBSCRIBED orders:* 3        ← 3 historical events incoming
SEVENT orders:1 101 {"total":49.99}
SEVENT orders:2 102 {"total":12.00}
SEVENT orders:3 105 {"total":8.50}
LIVE                         ← caught up, now streaming live
SEVENT orders:4 201 {"total":99.00}   ← new event arrives

# Client saves cursor: 201

# After reconnect — resume without missing anything
SUBSCRIBE orders:* FROM 201
SUBSCRIBED orders:* 0        ← nothing missed
LIVE
SEVENT orders:5 202 {"total":5.00}    ← continues from here

vs WATCH

WATCHSUBSCRIBE
Survives reconnectNo — events missed while offlineYes — replay from cursor
Event formatEVENT <stream> <payload>SEVENT <stream> <id> <payload>
History replayNoYes
Use whenEphemeral UI updates, presenceAny data that must not be lost

Notes

  • History is replayed from in-memory event log (bounded by max_history_per_stream). For very old events beyond the in-memory window, use SINCE to replay from the WAL first.
  • The token must have can_since = true (the default) to use SUBSCRIBE.
  • FROM 0 replays all available in-memory history.
  • Pattern wildcards work the same as WATCH — e.g. orders:*, *.

Durable Queues

SaturnDB has a built-in at-least-once job queue — no separate Redis or BullMQ needed. Jobs survive restarts (persisted in snapshots) and are automatically re-queued if a worker crashes before ACKing.

How it works

  1. Producer calls ENQUEUE <queue> <json> — server returns a unique job id.
  2. Worker calls DEQUEUE <queue> [timeout_secs] — claims the next job, making it invisible for timeout_secs (default 30s).
  3. Worker processes the job, then calls ACK <queue> <id> to permanently remove it.
  4. If the worker crashes or calls NACK, the job is returned to the front of the queue with an incremented retry counter.
  5. If the worker takes longer than timeout_secs, the job is automatically re-queued (checked every second).

Commands

text
# Producer
ENQUEUE jobs:email {"to":"user@example.com","subject":"Welcome"}
OK 1001                  ← server-assigned job id

# Worker
DEQUEUE jobs:email 60    ← claim for up to 60 seconds
JOB 1001 {"to":"user@example.com","subject":"Welcome"}

# Success — remove permanently
ACK jobs:email 1001
OK

# Failure — return to queue for retry
NACK jobs:email 1001
OK

# Inspect queue
QLEN  jobs:email         → COUNT 3   (pending + in-flight)
QPEEK jobs:email         → JOB 1002 {...}  (peek without claiming)

Long-poll DEQUEUE

Add WAIT [secs] to block until a job is available — or until the timeout expires. This eliminates polling loops in workers and lets the server push jobs the instant they arrive.

text
# Block up to 30 seconds (default) waiting for the next job
DEQUEUE jobs:email WAIT

# Block up to 10 seconds, claim with 60s visibility timeout
DEQUEUE jobs:email WAIT 10 60

# If a job arrives within the wait window:
JOB 1005 {"to":"user@example.com"}

# If no job arrives:
EMPTY

Long-poll DEQUEUE never blocks the connection — the server keeps it alive for WATCH events and disconnection during the wait. Multiple workers can all WAIT on the same queue and each gets a unique job.

Visibility Timeout

The optional timeout_secs argument to DEQUEUE controls how long a job stays invisible while a worker holds it. Defaults to 30 seconds. If the worker does not ACK or NACK within that window, the server automatically returns the job to the front of the queue.

Set the timeout to at least 2× your expected processing time. For long tasks, call NACK and re-DEQUEUE to reset the clock.

Persistence & Restart Safety

All pending jobs are saved in saturn-snapshot.json. In-flight jobs (claimed but not yet ACKed) are converted back to pending on restart — workers will re-process them. This guarantees at-least-once delivery.

Access Control

  • ENQUEUE, DEQUEUE, ACK, NACK — require write permission on the queue namespace.
  • QLEN, QPEEK — require read permission.
  • Queue names follow the same namespace:name convention as streams (e.g. jobs:email).

Geospatial Queries

SaturnDB has a built-in geo index that stores a latitude/longitude position per stream. Positions are indexed in memory and persisted in snapshots. Radius searches use the Haversine formula (Earth radius 6371 km) — no external dependency.

Auto-indexing

Any EMIT or PATCH whose JSON payload contains numeric lat and lng fields is automatically indexed. No extra command needed.

text
EMIT devices:42 {"lat": 48.8566, "lng": 2.3522, "status": "active"}
OK
# ↑ position is now in the geo index automatically

Commands

text
# Explicitly set a position
GEO SET devices:42 48.8566 2.3522
OK

# Get position
GEO GET devices:42
GEO_POS 48.8566 2.3522

# Radius search — all streams matching pattern within N km, sorted by distance
GEO NEAR devices:* 48.85 2.35 5.0
GEO_RESULT devices:42 48.8566 2.3522 0.083421
GEO_RESULT devices:7  48.8901 2.3100 4.218034
GEO_END

# Distance between two streams
GEO DIST devices:42 devices:7
GEO_DIST 4.218034

# Remove from geo index
GEO DEL devices:42
OK

Access Control

  • GEO SET / GEO DEL — requires write permission on the stream's namespace
  • GEO GET / GEO NEAR / GEO DIST — requires read permission
  • Auto-indexing on EMIT/PATCH uses the same auth check as the write itself — no extra grants needed

Persistence

Geo positions are saved in saturn-snapshot.json alongside stream data and survive server restarts. Positions are also removed from the index when a stream is expired or evicted.

Clustering

The geo index is per-node. GEO NEAR queries the local node only. In a clustered setup, emit to the shard that owns the stream — that node auto-indexes the position. For a globally consistent radius search across shards, query each node separately and merge client-side (scatter-gather for GEO NEAR is planned for V2).

Transactions

Transactions allow you to batch multiple writes into a single atomic operation. Commands inside a transaction return QUEUED. A client disconnect automatically triggers a rollback.

text
BEGIN
EMIT orders:1    {"item":"book","qty":2,"status":"confirmed"}
EMIT inventory:book {"stock":98}
EMIT audit:1     {"action":"order_placed","order":"orders:1"}
COMMIT

Use transactions any time you need multiple streams to stay in sync — e.g. order creation that simultaneously updates inventory and an audit log.

PATCH Operators

PATCH supports atomic field-level operators in addition to plain merging. A field whose value is a JSON object with only $-prefixed keys is treated as an operator — applied server-side with no read-modify-write round trip needed.

OperatorExampleEffect
$inc{"views": {"$inc": 1}}Add N to a numeric field. Creates the field at N if missing.
$mul{"price": {"$mul": 1.1}}Multiply field by N.
$min{"temp": {"$min": 0}}Set to the smaller of current and N.
$max{"score": {"$max": 100}}Set to the larger of current and N.
$push{"tags": {"$push": "vip"}}Append a value to an array field. Creates the array if missing.
$pop{"queue": {"$pop": "first"}}Remove from array. "first" or "last" (default).
$unset{"tmp": {"$unset": true}}Delete the field.
text
EMIT stats:user:42 {"rides": 0, "rating": 5.0, "tags": []}
OK

PATCH stats:user:42 {"rides": {"$inc": 1}}
OK
PATCH stats:user:42 {"rating": {"$min": 4.8}}
OK
PATCH stats:user:42 {"tags": {"$push": "frequent"}}
OK

GET stats:user:42
VALUE {"rides":1,"rating":4.8,"tags":["frequent"]}

Operators work inside transactions too — each PATCH returns QUEUED and all operations are applied atomically on COMMIT.

SDK Reference

All SDKs share an identical SaturnDBClient(host, port, token) API. Authentication is sent automatically on connect. All clients auto-reconnect with exponential backoff and re-subscribe to all WATCH patterns after reconnect.

MethodDescription
connect()Open TCP connection and authenticate
emit(stream, payload)Write an event to a stream
get(stream)Read the latest event from a stream
watch(pattern, handler)Subscribe to a pattern; handler called on every new event
since(stream, ts)Replay history from timestamp (Unix micros, 0 = all)
del(stream)Tombstone delete a stream
ping()Health check

Node.js SDK located at sdk/js/saturn.js. Uses raw TCP sockets with pipelining. Works in any Node.js environment.

javascript
const { SaturnDBClient } = require('./sdk/js/saturn.js');

const client = new SaturnDBClient('127.0.0.1', 7379, 'my-token');
await client.connect();

// Write an event
await client.emit('chat:1', { user: 'alice', text: 'hello world' });

// Read latest state
const msg = await client.get('chat:1');
console.log(msg); // { user: 'alice', text: 'hello world' }

// Subscribe to a pattern
await client.watch('chat:*', (stream, data) => {
  console.log(`[${stream}]`, data);
});

// Replay history from the beginning
const history = await client.since('chat:1', 0);
// history is an array of { stream, payload, ts } objects

// Clean up
await client.disconnect();

Python SDK located at sdk/python/saturn.py. Async/await based using asyncio. Requires Python 3.8+.

python
from saturn import SaturnDBClient
import asyncio

async def main():

    # Read latest state
    user = await client.get('users:1')

Dart/Flutter SDK located at sdk/dart/saturn.dart. Works on mobile, desktop, and web (via WebSocket on port 7380).

Architecture

SaturnDB is a single-binary server with no external dependencies. The core is a set of cooperating async actors communicating over tokio mpsc channels.

Module Overview

Event
event.rs
The core data structure: id, ts (Unix micros), stream, op (Set / Delete / Patch), payload.
Log
log.rs
Append-only WAL. Every write appends one JSON line to saturn.log. WAL actor uses mpsc + group commit: N concurrent EMITs → 1 syscall.
Index
index.rs
In-memory index: stream → Vec<Event>. O(1) reads. Holds expiry and deleted maps. Built from snapshot + WAL on startup.
Broker
broker.rs
Pub/sub fan-out. Pattern-aware — users:* matches all user streams. Each pattern has a tokio broadcast channel.
Protocol
protocol.rs
Command parser. Reads binary or text frames and produces a typed Command enum consumed by the handler.
Server
server.rs
TCP listener. One tokio task per client. Dispatches commands to log, index, and broker. Handles TLS or plain TCP based on config.
Auth
auth.rs
Token store. Validates roles, namespace access, and token expiry on every command.
Snapshot
snapshot.rs

WAL & Snapshots

Every write appends one JSON line to saturn.log. The WAL actor batches concurrent writes using a group commit: many parallel EMITs collapse into a single fsync (sync mode) or fire-and-forget (async mode).

Snapshots save the full in-memory index state to saturn-snapshot.json every N events. On startup, SaturnDB loads the snapshot first, then replays only the WAL entries that arrived after the snapshot — keeping startup fast even with large logs.

Async WAL

~55k ops/s

Fire-and-forget. Highest throughput. Best for most use cases where occasional data loss on hard crash is acceptable.

wal_mode = "async"

Sync WAL

Durable

Fsync on every write. Lower throughput but every committed event is on disk. Use for financial or auction data.

wal_mode = "sync"

Design Decisions

DecisionRationale
Append-only log Never mutate history. Current state = last event per stream. Makes time-travel and audit trivial.
No schema Any JSON payload, any stream name. Zero configuration to start writing data.
Stream naming convention namespace:id (e.g. users:123, chat:room1) enables namespace-scoped tokens and wildcard subscriptions.
Embedded WebSocket proxy Browsers connect on port 7380 — no separate process, no config. The same binary handles TCP and WS.
mimalloc allocator Global allocator override. Measurable throughput improvement over the system allocator under high concurrency.
TLS off by default Plain TCP for local dev. When tls = true, a self-signed cert is auto-generated — no manual cert setup.
